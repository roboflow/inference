Why The Gap Exists

The p50 total gap is 27.0 ms. Inference Server's measured RF-DETR postprocess alone is 21.9 ms, which explains most of the normal-case difference by itself.

Triton is not doing the same output work. It returns raw model tensors from a [1, 3, 384, 384] network input path, while Inference Server turns those outputs into API predictions and serializes a JSON response. That response is also parsed by the benchmark client.

A rough p50 sum of server-side surfaces absent or different in the Triton run - server preprocess, postprocess, request parse, response preparation, and client JPEG/base64 prepare - is 38.4 ms. These components overlap with end-to-end timers, so this is not an additive latency model, but it shows where the missing work lives.

TensorRT Is Not The Apparent Bottleneck

Inference Server model_predict is 10.0 ms p50 and stays near 10.3 ms at p99. Triton request p50 is 22.0 ms, which is 11.9 ms higher because it includes Triton HTTP/raw tensor transfer around execution.

The server tail is instead tied to RF-DETR postprocess: server max total latency is 469.8 ms, and server max postprocess is 438.5 ms. Triton max total latency is only 33.2 ms because it stops before that postprocess/API response path.

Interpretation Detail
Not apples-to-applesTriton returns raw tensors; Inference Server returns API predictions.
Output coordinate workInference Server postprocess maps raw RF-DETR outputs back into prediction objects.
Serialization pathInference Server includes JSON response materialization and client JSON parsing; Triton does not produce the same JSON payload.
Tail behaviorInference Server max latency tracks postprocess spikes; Triton max remains near 33 ms.
